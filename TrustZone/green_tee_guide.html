Project Overview Green TEE

What it is, why it was built, and what problems it solves

What is Green TEE?

Green TEE is a Trusted Operating System (Trusted OS) designed to run in Secure-EL1 (S-EL1) on ARMv8-A processors with TrustZone support. It is a learning-oriented, from-scratch implementation of a TEE that demonstrates the full stack: from TF-A boot to Linux kernel integration to userspace client applications.

The project currently supports three secure services:

GREEN_TEE_PRINT

Print a string in the Secure World terminal sent from a NS-EL0 application. Demonstrates cross-world data passing.

GREEN_TEE_ENCRYPT

Encrypt a buffer from NS-EL0 using the One Time Pad (OTP) algorithm inside the Secure World.

GREEN_TEE_DECRYPT

Decrypt a buffer from NS-EL0 using OTP inside the Secure World. Demonstrates secure key/data isolation.

What Problem Does It Solve?

Modern devices need to run security-sensitive code (key storage, biometric verification, DRM, cryptography) completely isolated from the general-purpose OS which may be compromised. Green TEE demonstrates how ARM TrustZone hardware isolation is leveraged to create this Secure World that Linux (Non-Secure World) cannot directly access or tamper with.

Why Build It When OP-TEE Exists?

OP-TEE is production-grade and complex. Green TEE is intentionally minimal and educational — it strips away GlobalPlatform APIs, complex scheduling, and thousands of lines of abstraction so you can see exactly what happens at each TrustZone boundary crossing.

ARM Architecture Features Used

ARMv8-A Exception Levels

EL0–EL3, S-EL0, S-EL1. Hardware-enforced privilege separation between all software layers.

TrustZone (NS Bit)

The SCR_EL3.NS bit separates Secure vs Non-Secure world at the hardware bus level — peripherals, memory regions, DRAM partitions.

SMC Instruction

Secure Monitor Call — the only architectural mechanism to cross from Non-Secure world to EL3 Secure Monitor (TF-A BL31).

TZASC / TZPC

TrustZone Address Space / Protection Controllers partition DRAM and peripherals between worlds. Only accessible from Secure state.

GICD / GICC (GICv2)

Interrupt controller with Secure vs Non-Secure interrupt routing. FIQ is typically routed to Secure World.

VBAR_EL3 / VBAR_EL1

Vector Base Address Registers — each world has its own exception vector table for handling IRQ, FIQ, SVC, SMC.

Component Details

Boot ROM

Burned into chip at manufacture. Runs at reset in EL3, Secure state. Initialises minimal hardware, verifies and loads BL1 from flash/eMMC. Cannot be modified. On QEMU, this is emulated and jumps directly to BL1 entry.

BL1 — Boot Loader Stage 1

Runs at EL3, Secure. Initialises exception vectors for EL3, sets up SRAM, loads BL2 image and verifies it (in production — QEMU skips signature check). Hands off to BL2 via BL1_RUN_IMAGE SMC.

BL2 — Boot Loader Stage 2

Runs at S-EL1 (or EL1 Secure). Loads all remaining images: BL31, BL32 (Green TEE), BL33 (U-Boot). Sets up memory descriptors and passes them to BL31 via bl_params structure.

BL31 — EL3 Runtime Firmware

The permanent EL3 runtime. Initialises the Secure Monitor, registers runtime services (PSCI, SiP, Standard, green_teed SPD). After init, drops to BL33 (U-Boot) in NS-EL1. From that point on, BL31 only runs when an SMC occurs.

green_teed — Secure Payload Dispatcher (SPD)

A TF-A runtime service registered at EL3. When the Linux kernel driver issues an SMC with Green TEE's SMCCC function ID, BL31 dispatches to green_teed which saves NS context, restores S-EL1 context, and switches to Green TEE. This is the cross-world gate.

Green TEE Core (S-EL1)

The Trusted OS. Has its own exception vectors, MMU, stack, and service dispatch table. Receives requests from green_teed, processes them (PRINT/ENCRYPT/DECRYPT), and SMC-returns to EL3 which restores NS context.

Linux Kernel TEE Driver

A character device driver at drivers/green_tee/. Exposes /dev/green_tee. User applications call ioctl() → driver marshals the request into SMC registers (x0–x7) → issues smc instruction → blocks until EL3 returns.

Userspace Client

A C application in client/. Opens /dev/green_tee, calls ioctl(ENCRYPT, buf) or ioctl(PRINT, "hello"). This triggers the full call chain down to S-EL1 and back.

Boot Flow Step-by-step

From power-on to Linux prompt — every stage with EL, security state, and software

1

Power ON / Reset

EL3 Secure Boot ROM / QEMU reset vector

CPU resets to EL3, Secure state. All registers are in reset state. QEMU initialises the virtual machine and jumps to the TF-A BL1 entry point at 0x0e000000 (or wherever BL1 is loaded in flash). On real hardware, the Boot ROM verifies BL1 signature before executing.

2

BL1 — Boot Loader Stage 1

EL3 Secure arm-trusted-firmware/bl1/

BL1 sets up EL3 exception vectors, initialises SRAM/caches, detects the SoC. It then loads the BL2 image from flash into SRAM and transfers control via bl1_run_next_image() which does an ERET to BL2 in S-EL1 (or EL1 Secure on older configs).

/* BL1 key actions */
bl1_early_platform_setup();   /* init UART, clocks */
bl1_plat_arch_setup();        /* setup MMU for BL1 */
bl1_load_bl2();               /* copy BL2 from FIP/flash */
bl1_run_next_image(&bl2_ep);  /* ERET to S-EL1 */
3

BL2 — Image Loader

S-EL1 Secure arm-trusted-firmware/bl2/

BL2 runs in S-EL1. Its job is to load all firmware images needed by BL31. It reads the Firmware Image Package (FIP) — a single binary containing BL31, BL32 (Green TEE), and BL33 (U-Boot). It populates a bl_params linked list describing where each image is loaded in memory.

After loading all images, BL2 issues BL1_RUN_IMAGE (SMC 0) to return to BL1, which then passes control to BL31.

4

BL31 — EL3 Runtime Firmware Init

EL3 Secure arm-trusted-firmware/bl31/

BL31 is the permanent EL3 resident. It initialises:

  • EL3 exception vectors (VBAR_EL3)
  • Runtime services: PSCI (power), SiP, Standard, green_teed SPD
  • Context management: NS context and S-EL1 context structures
  • GIC initialization for secure interrupt routing

After init, BL31 calls green_teed_init() which drops to S-EL1 to initialise Green TEE. Green TEE init returns via SMC, BL31 then does ERET to NS-EL1 to start U-Boot.

5

Green TEE Initialisation (S-EL1)

S-EL1 Secure tee/

BL31 switches to S-EL1 and jumps to Green TEE's entry point. Green TEE initialises:

  • Its own exception vector table (VBAR_EL1 in Secure state)
  • Secure UART (forwarded to QEMU port 12345)
  • Service dispatch table (PRINT, ENCRYPT, DECRYPT)
  • Secure memory regions

Green TEE signals completion by issuing an SMC back to EL3.

6

U-Boot (NS-EL1)

NS-EL1 Non-Secure u-boot/

BL31 does ERET to NS-EL1 with U-Boot's entry point. U-Boot initialises DDR, QEMU virtio block device, loads the Linux kernel image + DTB + initramfs from the buildroot image. Then executes booti to hand off to Linux.

7

Linux Boot → Runtime

NS-EL1 Non-Secure linux/ + drivers/green_tee/

Linux boots normally in NS-EL1. During driver init, the green_tee driver registers /dev/green_tee character device. When the client app calls ioctl(), the driver issues an SMC — this is the first runtime cross-world call to Green TEE. The system is now fully operational with both worlds running.

TrustZone Concepts Core Theory

Every fundamental concept you need to understand Green TEE's operation

Secure World vs Non-Secure World

ARM TrustZone extends hardware with a single control bit: SCR_EL3.NS (Non-Secure bit in the Secure Configuration Register at EL3). When NS=0, the CPU is in Secure state. When NS=1, Non-Secure state. This single bit propagates to the AXI bus as the HNONSEC/AWPROT[1] signal, which TZASC uses to allow or deny memory access.

Key insight: The hardware enforces world separation — even if Linux kernel code tries to read secure DRAM, the TZASC raises a bus fault. Software cannot bypass this without going through EL3.

World Switching Mechanism

The only way to switch worlds is through EL3. Here is the complete SMC call flow:

/* Step 1: NS-EL1 Linux kernel driver issues SMC */
mov x0, #GREEN_TEE_SMC_FID   /* Function ID in x0 */
mov x1, #SERVICE_ENCRYPT      /* Service type */
mov x2, #buf_addr             /* Shared memory address */
mov x3, #buf_size
smc #0                        /* Trap to EL3 */

/* Step 2: EL3 (BL31) receives exception via VBAR_EL3[SMC64] vector */
/*         SMC handler calls green_teed_smc_handler() */
/*         green_teed saves NS context (x0-x29, SP_EL1, ELR_EL3, SPSR_EL3) */
/*         Restores S-EL1 context */
/*         Sets SCR_EL3.NS = 0 */
/*         ERET to Green TEE's handler in S-EL1 */

/* Step 3: Green TEE (S-EL1) processes request, issues SMC to return */
smc #0                        /* Return to EL3 */

/* Step 4: EL3 restores NS context, sets SCR_EL3.NS = 1, ERET to NS-EL1 */
/* Linux driver gets return values in x0-x3 */

Context Switching (NS ↔ S)

The Secure Monitor (BL31) maintains two complete CPU contexts. When switching worlds, it must save/restore:

General Purpose Registers

x0–x30, SP_EL0, SP_EL1 for each world. x0–x7 are used for SMC argument passing (SMCCC convention).

System Registers

ELR_EL3 (exception link — return address), SPSR_EL3 (saved program state), SCTLR_EL1, TTBR0_EL1/TTBR1_EL1 (page tables per world), MAIR_EL1, TCR_EL1.

Security Configuration

SCR_EL3.NS is set/cleared. HCR_EL2 may be modified. DAIF interrupt masks are managed per world.

SMC Calling Convention (SMCCC)

ARM defines a standard SMC calling convention (SMCCC v1.x) for how parameters are passed:

/* SMCCC register usage for Green TEE */
x0  = Function ID (encodes: call type, service owner, function number)
x1  = Argument 1 (service type: PRINT/ENCRYPT/DECRYPT)
x2  = Argument 2 (shared memory physical address)
x3  = Argument 3 (buffer length)
x4-x7 = Additional args (if needed)

/* Function ID encoding (32-bit) */
/* [31]    = 0 (SMC32) or 1 (SMC64)             */
/* [29:24] = Owning entity (0x32 = Trusted OS)  */
/* [15:0]  = Function number                    */

Interrupt Routing (FIQ vs IRQ)

In TrustZone systems, interrupts are split:

  • FIQ (Fast Interrupt Request) → Routed to Secure World (EL3 or S-EL1). Used for Secure peripherals, Trusted Watchdog, Secure timer.
  • IRQ (Interrupt Request) → Routed to Non-Secure World (NS-EL1). Used by Linux for all normal device interrupts.
  • SCR_EL3.FIQ=1 means FIQ traps to EL3 even if running in NS world — the Secure Monitor decides whether to forward or handle.

Source Code Walkthrough File-by-file

Every directory and key file explained with its role in the system

tee/ — Green TEE Core (S-EL1 Trusted OS)

This is the heart of the project. It's a small bare-metal OS running at S-EL1.

tee/src/entry.S — Startup Assembly

The very first code that runs in S-EL1. Sets up the stack pointer, initialises BSS section, sets VBAR_EL1 to the exception vector table, then calls main() in C. This is the Green TEE equivalent of Linux's head.S.

/* Conceptual structure of entry.S */
_start:
    adr  x0, __stack_top
    mov  sp, x0            /* Setup stack */
    bl   zero_bss          /* Clear BSS */
    adr  x0, exception_vectors
    msr  vbar_el1, x0      /* Set exception vectors */
    bl   tee_main          /* Jump to C entry */
    b    .                 /* Should never return */

tee/src/vectors.S — Exception Vector Table

The 2KB-aligned exception vector table for S-EL1. ARM defines 16 vectors (4 sources × 4 types). For Green TEE, the most critical is the Synchronous exception from lower EL using AArch64 — this fires when EL3 does ERET into Green TEE passing an SMC request.

/* ARM exception vector layout (each entry 128 bytes) */
.align 11                       /* 2KB alignment required */
exception_vectors:
    /* Current EL with SP0 */
    .align 7; b sync_sp0        /* +0x000: Sync */
    .align 7; b irq_sp0         /* +0x080: IRQ */
    .align 7; b fiq_sp0         /* +0x100: FIQ */
    .align 7; b serr_sp0        /* +0x180: SError */
    /* Current EL with SPx */
    ...
    /* Lower EL AArch64 — this is where SMC requests arrive */
    .align 7; b tee_entry_from_el3  /* +0x400: Sync lower AArch64 */

tee/src/service.c — Service Dispatcher

Reads the service type from x1 (passed via SMC registers), dispatches to the correct handler. This is the "syscall table" of the Trusted OS.

void tee_handle_request(tee_args_t *args) {
    switch (args->service_id) {
        case GREEN_TEE_PRINT:
            tee_service_print(args->buf, args->len);
            break;
        case GREEN_TEE_ENCRYPT:
            tee_service_encrypt(args->buf, args->len);
            break;
        case GREEN_TEE_DECRYPT:
            tee_service_decrypt(args->buf, args->len);
            break;
        default:
            args->ret = TEE_ERR_UNKNOWN_SERVICE;
    }
}

tee/src/crypto.c — One Time Pad Crypto

Implements OTP encryption/decryption. XORs each byte of the input buffer with a key byte (modulo key length). This runs entirely inside S-EL1 — neither the key nor the plaintext ever exists in Non-Secure memory during processing.

void otp_encrypt(uint8_t *buf, size_t len) {
    const uint8_t key[] = {0xDE, 0xAD, 0xBE, 0xEF}; /* Secure key in S-EL1 */
    for (size_t i = 0; i < len; i++)
        buf[i] ^= key[i % sizeof(key)];
}

arm-trusted-firmware/services/spd/green_teed/ — SPD

This is the EL3 "plugin" that TF-A calls when a Green TEE SMC arrives.

green_teed_main.c — SPD Core

Registers Green TEE's runtime service with BL31's service framework. Implements green_teed_smc_handler() — the function TF-A calls for every SMC with Green TEE's owner ID. This function:

  • Saves all NS CPU context (registers, system regs) into ns_cpu_context
  • Restores S-EL1 context from tee_cpu_context
  • Calls cm_prepare_el3_exit(SECURE) to set SCR_EL3.NS=0
  • Returns tee_entry_point — BL31 will ERET to Green TEE

linux/drivers/green_tee/ — Kernel Driver

green_tee_drv.c — Character Device Driver

A standard Linux misc device driver. Registers /dev/green_tee. The unlocked_ioctl handler is the bridge:

static long green_tee_ioctl(struct file *f, unsigned int cmd, unsigned long arg) {
    struct green_tee_req req;
    copy_from_user(&req, (void __user*)arg, sizeof(req));

    /* Issue SMC to EL3 → green_teed SPD → Green TEE */
    struct arm_smccc_res res;
    arm_smccc_smc(GREEN_TEE_SMC_FID, cmd, req.buf_pa, req.len,
                   0, 0, 0, 0, &res);

    copy_to_user((void __user*)arg, &req, sizeof(req));
    return res.a0; /* Return value from Green TEE */
}

client/ — Userspace Application

client/main.c — Test Application

Opens /dev/green_tee, allocates a shared buffer, populates it, and calls ioctl() with the desired service. After the call returns, reads the (potentially encrypted) result. This is the NS-EL0 application that drives all TEE requests.

int fd = open("/dev/green_tee", O_RDWR);
struct green_tee_req req = {
    .buf  = (uint8_t*)"hello secure world",
    .len  = 18
};
ioctl(fd, GREEN_TEE_PRINT, &req);    /* Print in Secure terminal */
ioctl(fd, GREEN_TEE_ENCRYPT, &req);  /* Encrypt in S-EL1 */

Linker Scripts

Green TEE has its own linker script defining the secure memory layout. Key sections:

MEMORY {
    SECURE_ROM  (rx)  : ORIGIN = 0x06000000, LENGTH = 256K  /* TEE code */
    SECURE_RAM  (rwx) : ORIGIN = 0x06040000, LENGTH = 512K  /* TEE data/stack/heap */
}
SECTIONS {
    .text    : { *(.text*) } > SECURE_ROM
    .rodata  : { *(.rodata*) } > SECURE_ROM
    .data    : { *(.data*) } > SECURE_RAM AT> SECURE_ROM
    .bss     : { *(.bss*) } > SECURE_RAM
    .stack   : { . = ALIGN(16); . += STACK_SIZE; } > SECURE_RAM
}

Execution Environment Where to run it

Compatibility analysis across every platform you might use

PlatformWorks?Notes
QEMU (aarch64-softmmu) ✅ Primary target Full TrustZone emulation with QEMU virt board. Both Secure and NS UART forwarded. This is the intended platform.
Ubuntu x86_64 laptop ✅ Build host You build everything here using aarch64 cross-compiler. Run via QEMU. You never run Green TEE natively on x86 — it's an ARM-only TEE.
Raspberry Pi 4 ⚠ Possible with effort RPi4's BCM2711 has TrustZone but the bootloader (RPi firmware) is closed-source. You'd need to replace the boot chain with TF-A + UEFI (done by the RPi TF-A port). Non-trivial but doable.
Qualcomm boards ❌ Not recommended Qualcomm locks the TrustZone. Their QSEE uses signed SPDs. You cannot load an unsigned Trusted OS. Development boards (RB3, RB5) still have QSEE locked.
HiKey 960 🔵 Potentially 96Boards platform with open TF-A support. OP-TEE runs on it. Green TEE could be ported, but would require platform-specific UART, GIC, and memory map adaptation.
ARM64 VM (AWS Graviton etc.) ❌ No Cloud VMs do not expose TrustZone to guest VMs. You'd need bare-metal ARM64 with Type-1 hypervisor support (KVM/ARM does have TrustZone passthrough research, but not general availability).

Recommendation: Use QEMU on Ubuntu x86_64. It's exactly what the project is designed for and gives you full visibility into both worlds with two separate terminal windows.

NS-EL0Non-Secure

/* CPU: NS-EL0 | syscall (SVC #0) → NS-EL1 */

The SVC instruction causes a synchronous exception trapping into NS-EL1 Linux kernel. No world switch yet — this is a normal Linux syscall.

NS-EL1Non-Secure
/* drivers/green_tee/green_tee_drv.c */
green_tee_ioctl():
    copy_from_user(&req, user_ptr, sizeof(req))
    /* Prepare SMC arguments */
    x0 = GREEN_TEE_SMC64_FID      /* 0xC2000001 (example) */
    x1 = GREEN_TEE_ENCRYPT        /* service type */
    x2 = virt_to_phys(req.buf)    /* physical address! */

    /* SMC instruction executes → EL3 */

The driver converts the virtual address to physical (Green TEE's MMU may have a different mapping) and issues the SMC. The CPU takes a synchronous exception to EL3.

TrustZone Fundamentals

What is the NS bit and where is it set? TrustZone▶
The NS (Non-Secure) bit is bit 0 of SCR_EL3 (Secure Configuration Register). When NS=1, the processor is in Non-Secure state; NS=0 = Secure state. Only EL3 code can modify SCR_EL3. This bit also propagates to the AXI/ACE bus as the HNONSEC signal, which TZASC uses to allow or deny memory access.
How does a world switch happen? Can Linux trigger it directly? TrustZone▶
A world switch ONLY happens via the Secure Monitor (EL3). Linux (NS-EL1) issues an SMC instruction which traps to EL3. The EL3 monitor saves NS context, optionally switches to Secure world (restoring S-EL1 context), and does ERET. Linux cannot directly set SCR_EL3.NS — that register is only accessible from EL3.
What is TZASC and how does it protect memory? TrustZone▶
TZASC (TrustZone Address Space Controller) is a hardware component on the memory bus (between CPU and DRAM controller). It is configured by Secure world software during boot to mark DRAM regions as Secure or Non-Secure. When a Non-Secure bus transaction (NS=1) attempts to access a Secure-marked region, TZASC raises a DECERR (decode error) bus fault — the access is denied in hardware, independent of software.
What is the difference between FIQ and IRQ in TrustZone? TrustZone▶
FIQ (Fast IRQ) is typically configured to route to Secure World (EL3 or S-EL1 depending on SCR_EL3.FIQ). IRQ routes to Non-Secure World. This allows the Secure World to have priority interrupts (Trusted Watchdog, secure timer) that Linux cannot mask. SCR_EL3.IRQ=1 routes IRQs to EL3 too (used in some configurations for interrupt virtualization).
Can Secure world access Non-Secure memory? TrustZone▶
Yes, typically. TZASC protection is usually one-directional: Non-Secure bus transactions cannot access Secure regions, but Secure transactions can access Non-Secure regions. However, this is configurable — TZASC can optionally block Secure→NS as well. In Green TEE, the TEE accesses shared NS memory (passed as a PA via SMC) to read/write the user's buffer.
What registers must be saved during a world switch? TrustZone▶
Full context includes: General-purpose registers x0–x30, SP_EL0, SP_EL1. System registers per EL1: SCTLR_EL1, TTBR0_EL1, TTBR1_EL1, TCR_EL1, MAIR_EL1, AMAIR_EL1, VBAR_EL1, ELR_EL1, SPSR_EL1, CONTEXTIDR_EL1. Plus EL3 return state: ELR_EL3, SPSR_EL3. SIMD/FP registers (v0–v31, FPSR, FPCR) must also be saved if the TEE uses floating point.

TF-A Architecture

What is BL31 and when does it run? TF-A▶
BL31 is the EL3 Runtime Firmware — it is the only TF-A component that stays resident in memory permanently (loaded into SRAM/secure DRAM). It runs whenever an SMC is issued from any EL below EL3. BL31 initialises once at boot (setting up exception vectors, runtime services, context management), then transitions to U-Boot/Linux. After that, it only runs on SMC exceptions.
What is a Secure Payload Dispatcher (SPD)? TF-A▶
An SPD is a TF-A plugin that manages a specific Secure OS (BL32). It registers with BL31's runtime service framework and receives SMC calls for its owner ID. The SPD is responsible for: initialising the Trusted OS during boot, saving/restoring NS and Secure contexts on every SMC, and dispatching appropriate requests to the Trusted OS. green_teed is Green TEE's SPD.
What is a FIP (Firmware Image Package)? TF-A▶
A FIP is a single binary container (like a tar archive) that packages all TF-A firmware images: BL1, BL2, BL31, BL32 (TEE), BL33 (U-Boot/UEFI), certificates, and device tree. BL2 knows how to parse the FIP format to extract and load each image. This simplifies the boot media — one file instead of many separate binaries at fixed offsets.
What is the difference between BL32 and BL31? TF-A▶
BL31 is the EL3 Secure Monitor — it runs at highest privilege, manages context switching, and is always resident. BL32 is the Secure OS payload (Trusted OS like Green TEE, OP-TEE) running at S-EL1. BL31 loads and initialises BL32 during boot, then manages all BL32 context switches via the SPD plugin. BL32 never runs at EL3 — it runs at S-EL1.
How does TF-A's context management API work? TF-A▶
cm_el1_sysregs_context_save(world) stores EL1 system registers for the specified world. cm_el1_sysregs_context_restore(world) loads them back. cm_set_next_eret_context(world) configures ELR_EL3 and SPSR_EL3 for the ERET to the specified world. These are the three core operations in every SPD SMC handler. TF-A stores contexts in per-CPU data structures.

FF-A & Secure Partitions

What is FF-A and what problem does it solve? FF-A▶
FF-A (Firmware Framework for Arm, DEN0077) standardises how software components (VMs, Secure Partitions, hypervisors) communicate across privilege levels in an ARMv8/v9 system. Before FF-A, each TEE had its own proprietary SMC interface. FF-A provides: standard message passing (FF-A messages), memory sharing descriptors, partition discovery, and a standard SPMC interface — enabling interoperability between multiple TEEs and hypervisors.
What is SPMC and SPMD? FF-A▶
SPMC (Secure Partition Manager Core) runs at S-EL2 (or S-EL1 in AArch64 without EL2). It manages Secure Partitions — isolated execution environments in the Secure World. SPMD (Secure Partition Manager Dispatcher) is the EL3 component in TF-A that acts as the relay between NS world and the SPMC. SPMD routes FF-A calls to SPMC; SPMC manages SP lifecycle and memory. Green TEE does NOT use FF-A — it uses a simpler direct SPD approach.
What is a Secure Partition (SP)? FF-A▶
A Secure Partition is an isolated execution environment managed by the SPMC, running in Secure World (S-EL0 or S-EL1). Each SP has its own virtual address space, interrupt assignment, and communication endpoints. SPs communicate via FF-A messages (direct request/response or indirect queued messages). Examples: OP-TEE as an SP, Trusted Services (crypto, attestation) as individual SPs at S-EL0.

Green TEE Specific

What exception level does Green TEE run at? Green TEE▶
Green TEE runs at S-EL1 (Secure Exception Level 1). It is the Secure OS / BL32 payload. It has its own exception vectors (VBAR_EL1 in Secure state), its own page tables (TTBR0/1_EL1 in Secure state), and its own stack. It is managed by the green_teed SPD running at EL3.
How does the Linux client communicate with Green TEE? Green TEE▶
The chain is: userspace app → ioctl(/dev/green_tee) → Linux TEE driver (drivers/green_tee/) → arm_smccc_smc() → EL3 BL31 → green_teed_smc_handler() → context switch → ERET to S-EL1 → Green TEE service handler → SMC return → EL3 restores NS → Linux driver returns → ioctl returns to app.
Why does Green TEE use One Time Pad for crypto? Green TEE▶
OTP is used for simplicity — it's a XOR operation, trivial to implement in a few lines without needing AES hardware acceleration or software libraries. The educational goal is to demonstrate that the KEY remains in Secure World memory (S-EL1 SRAM), unreachable by Linux, and that the processing happens isolated from the Normal World. In production, you'd use AES-GCM or ChaCha20-Poly1305.
What is green_teed and how is it different from Green TEE itself? Green TEE▶
green_teed (the "d" suffix means "dispatcher") is the TF-A EL3 plugin — it lives inside arm-trusted-firmware/services/spd/green_teed/ and runs at EL3 privilege. Green TEE (the tee/ directory) is the actual Trusted OS running at S-EL1. green_teed is the gateway: it receives SMCs at EL3 and orchestrates the context switch that hands off to Green TEE at S-EL1. They are separate binaries compiled into the FIP.
Why does the QEMU command need -machine virt,secure=on? Green TEE▶
The secure=on flag tells QEMU's virt machine to enable TrustZone emulation: it adds a TZASC (to enforce Secure memory access control), routes Secure UART to a separate serial port, and allows EL3 to function properly. Without this, QEMU doesn't emulate the NS bit, TZASC, or secure memory protection — the Secure Monitor would still run but there would be no actual memory isolation enforcement.

You're developing or testing the SPD/SPMC framework itself. TSP is TF-A's own reference BL32 with minimal services for regression testing the EL3→S-EL1 path.
